07 · CLAUDE CLI

claude 를 어떻게 굴리나

SDK 도, PTY 도 아닙니다. 호스트 바이너리를 그대로 spawn 해서 stdout 을 줄 단위로 끊고, 각 줄을 JSON 으로 파싱해 이벤트로 흘립니다. 단순하지만 — 그래서 안 깨집니다.

// claudeProcess.ts

import { spawn } from 'node:child_process'; const proc = spawn('claude', [ '--model', model, '--output-format', 'stream-json', '--verbose', '-p', prompt, '--resume', claudeSessionId, // 2턴부터 ], { cwd: sessionWorkDir, // 세션 폴더 stdio: ['ignore', 'pipe', 'pipe'], env: { ...process.env, FORCE_COLOR: '0', // 색 차단 CLAUDE_CODE_*: '', }, }); // stdout: 줄단위 JSON let buf = ''; proc.stdout.on('data', chunk => { buf += chunk; const lines = buf.split('\n'); buf = lines.pop()!; // 부분 줄 보관 for (const line of lines) { emitter.emit('data', JSON.parse(line)); } });

// 핵심 결정

  • PTY 아님 — node-pty 도, tmux 도 안 씁니다. 단순 spawn + stdio: pipe. claude 가 TTY 가 없어도 -p + stream-json 모드로 깔끔히 출력.
  • 줄 단위 line splitter — chunk 가 줄 중간에서 잘릴 수 있어, 마지막 부분 줄(buf)을 보관 후 다음 chunk 에 이어붙입니다.
  • 각 줄 = 하나의 JSON 이벤트 — tool_use, text, thinking, result 같은 type 필드로 구분. 그대로 WebSocket 으로 broadcast.
  • 색·진행바 차단 — FORCE_COLOR=0 + CLAUDE_CODE_* env 를 모두 비워 ANSI 코드가 stream-json 을 깨뜨리지 않게.
  • abort = SIGTERM → 5s → SIGKILL — 사용자 중단 시 우아하게 끄려 시도, 안 죽으면 강제.
  • 2 턴부터 --resume — 첫 prompt 는 새 세션, 그 뒤로는 claude 가 발급한 sessionId 로 이어가기.
SIMPLE
SDK 도, 자체 chat-loop 도 없습니다. CLI 한 줄 위에 얇은 line-parser 만 — 깨질 데가 없는 게 미덕.
08 · SESSION ISOLATION

세션마다 자기 작업장

사용자 A 의 작업이 사용자 B 에게 새지 않으려면 — 파일 시스템부터 갈라야 합니다. 세션이 생기면 git clone, 끝나면 통째로 rm -rf. 사용자 개인 작업 폴더는 절대 안 만집니다.
~/Projects/.sessions/ ├── a7f3-c821/ # 세션 1 │ └── apply-cert/ │ ├── src/... │ ├── apply-docs/ # submodule auto-init │ └── .git/ │ └── hooks/ │ └── pre-push # 자동 설치 ├── b9d2-ee15/ # 세션 2 — 다른 사용자 │ └── apply-api-gateway/ │ └── ... └── c4e8-7733/ # viewer 세션 (read-only) └── apply-cert/ └── CLAUDE.md # role-aware 자동 주입
📁
세션마다 새 clone — ~/Projects/<projectName> 의 사용자 작업 폴더와 완전히 분리. 한 사람의 미커밋 변경이 다른 세션에 새지 않습니다.
⎘
submodule 자동 init — apply-docs 같은 공통 의존도 자동으로 끌어와 remote 최신 상태로 update.
↑
pre-push hook 자동 설치 — push 직전 submodule sync 까지 알아서. AI 가 잊어도 hook 이 챙깁니다.
✕
세션 종료 / 서버 재시작 시 rm -rf — 작업 폴더 leak 이 0. 다시 시작해도 깨끗한 상태에서 출발.
📜
role-aware CLAUDE.md 자동 주입 — viewer 는 fs 쓰기 금지, user 는 schema/infra/.env 금지. 세션 폴더 루트에 작성돼 모델이 항상 읽고 시작.
ISOLATION
격리는 정책이 아니라 디렉토리 구조 로 합니다 — 셀이 따로 있으면, 사고는 그 셀 안에서 멈춥니다.

10 · CONTEXT PIPELINE

한 줄을 6 겹 컨텍스트 로

사용자가 "버튼 색 바꿔줘" 한 줄을 던지면, 메이커가 그 위에 여섯 겹의 컨텍스트 를 자동으로 합성해 claude 에게 넘깁니다. AX 의 진짜 속살 — 여기가 메이커를 메이커답게 만드는 자리입니다.
<persona>
역할별 인격 부여 — config.persona 에서 role 에 맞는 페르소나 텍스트 합성. user 면 "동료처럼", viewer 면 "안내자처럼".
<user-context>
누가 묻는지 — DB 에서 이름·역할·현재 프로젝트·접근 가능한 프로젝트 목록을 끌어와 첨부. "박영진(admin) · apply-cert"
<previous-conversations>
최근 5 개 대화 로그 — conversation_log 테이블에서. 새 세션이라도 이전 맥락을 자연스럽게 이어가게.
<attached-files>
업로드 첨부 — multipart 로 받아둔 파일들에 대해 "Read 도구로 확인하세요" 라고 명시. 모델이 자동으로 fetch.
<jabis-ecosystem-context>
핵심 — ContextResolver 가 프롬프트 텍스트에서 oauth·스키마·helm 같은 키워드를 detect → 관련 jabis-* 프로젝트의 docs/CLAUDE.md/API spec/DB 스키마를 토큰 예산 안에서 자동 첨부.
<suggestion-instruction>
후속 질문 3 개 를 요청 — 답변 끝에 [SUGGESTIONS]A|B|C[/SUGGESTIONS] 형식으로. 클라이언트는 이걸 잘라서 칩으로 뿌립니다.
사용자 입력 한 줄 + 6 layers → claude 에 한 번에 합성 전달
AX CORE
모델이 똑똑한 게 아니라 — 컨텍스트가 똑똑한 것입니다. ContextResolver 가 메이커의 진짜 엔진.
11 · INTERCEPT

claude 의 질문을 가로채다

claude 가 사용자에게 묻고 싶을 때 쓰는 AskUserQuestion 툴. 보통은 터미널에서 라디오 UI 로 뜨지만 — 메이커는 stream 에서 이걸 가로채서 웹 UI 의 라디오·체크박스로 변환 합니다.
STEP 01
claude 가 질문 시도
stream 에 tool_use: AskUserQuestion 이벤트 출력. 옵션 배열·multiSelect 여부 포함.
STEP 02
즉시 프로세스 kill
line-parser 가 이 이벤트를 감지하는 순간 proc.kill() — claude 가 sandbox 답변을 자동 생성하기 전에 끊습니다.
STEP 03
user_question 푸시
WebSocket 으로 user_question 메시지를 클라이언트에 broadcast. 옵션을 그대로 들고감.
STEP 04
웹 UI 에서 답변
사용자가 라디오/체크박스로 클릭. 클라이언트가 answer_question ws 메시지로 다시 서버에.
STEP 05
--resume 으로 재개
서버가 답변을 합성해 새 prompt 로 --resume spawn. claude 는 자기가 물어본 질문에 답이 돌아왔다고 인식.
UX BRIDGE
CLI 의 대화형 질문이 웹의 라디오·체크박스 로 — 같은 모델, 다른 인터페이스. 비개발자도 자연스럽게.
12 · EXEC MODES

한 모델, 네 가지 실행 패턴

같은 claude 를 굴려도 — 일하는 방식은 모드별로 다릅니다. 모드마다 전용 executor 클래스 가 있어 phase·체크리스트·quality gate 같은 진행 상황을 ws 로 broadcast 합니다.
MODE 01
default
interactive
일반 멀티턴 — --resume 으로 이어갑니다. 사용자 ↔ claude 가 한 prompt 씩 주고받는 클래식 패턴. 대부분의 작업이 이 모드.
multi-turn --resume stream
MODE 02
UltraExecutor
ultra_execute
페이즈·체크리스트 기반 단계 실행. plan → 분석 → 구현 → 검증 같은 phase 가 순차로 돌고, 각 단계에서 ws 로 ultra_phase · ultra_checklist 진행률 broadcast.
phased checklist live progress
MODE 03
NightExecutor
night_start
야간 자동화 배치. 여러 task 를 줄세워 돌리며 — quality gate 를 통과해야 다음 task. 실패 task 는 재시도 큐로, 통과한 변경은 push 까지 자동.
batch quality gate auto push
MODE 04

13 · GUARDRAILS

구현 레벨 가드레일

하네스 엔지니어링(정책 레벨)이 "어떤 길을 깔까" 였다면, 여기는 코드로 박힌 가드레일. JWT, 디바이스 토큰, 3-tier 권한, 세션별 CLAUDE.md, 자동 폴백까지 — 운영하면서 신경 쓴 것들.
G · 01
JWT 액세스 토큰
로그인 시 발급된 JWT 를 HttpOnly 쿠키 + ws 첫 auth 메시지에 동시에 들어가게. 만료되면 ws 가 자동 단절.
G · 02
디바이스 토큰
admin SPA 는 Vault-style 시크릿 을 1 회 입력해 활성화한 디바이스에서만 접근 가능. 토큰은 SQLite 에 hash 로 저장.
G · 03
3-tier role
admin · user · viewer. role 에 따라 ws 의 17 종 client 메시지 중 WRITE_ACTIONS 셋에 들어가면 자동 거부.
G · 04
세션 CLAUDE.md
세션 폴더 루트에 role 별 룰북 자동 작성. viewer = fs 쓰기 금지, user = schema/infra/.env 금지. 모델이 시작 시 읽고 따름.
G · 05
세션 소유권 체크
canUserAccessSession() 이 모든 ws 요청 앞단 에서 호출. 본인 소유가 아니면 attach/prompt 다 거부.
G · 06
Resume 자동 폴백
--resume 이 No conversation found 로 실패하면 자동으로 새 세션으로 재시작. claude 캐시 정리 후에도 사용자는 모름.
G · 07
rate limit
동일 user 당 동시 실행 세션·분당 prompt 횟수 제한. claude 가 무한루프성 작업에 빠져도 비용·시간이 폭주하지 않도록.
G · 08
Multipart upload cap
첨부 파일 size·count 상한. claude 가 거대 파일을 통째로 Read 하다가 컨텍스트 폭발하는 사고 차단.
G · 09
git token scope 제한
세션 폴더에 주입되는 git token 은 apply-* · feature/* 범위만. main 보호 + org admin 차단 — 슬라이드 5 에서 본 정책의 구현부.
14 · EXTRAS

그 밖에 붙어 있는 것들

본체 외에 — 메이커가 "그냥 굴러간다" 가 아니라 "꾸준히 굴러간다" 가 되게 해주는 잡다한 기능들. 발표 시간이 남으면 한 마디씩 짚을 자리입니다.
EX · 01
reportGenerator
일·주간 작업 리포트를 claude 가 자동 생성. 어떤 세션이 무엇을 만들었는지, 어디서 멈췄는지를 한 페이지로 요약 → 이메일·Teams 자동 발송.
scheduler daily / weekly
EX · 02
EmailNotifier
작업 완료·실패·quality gate 통과 이벤트를 메일로. 사용자가 night 모드를 걸어두고 자도, 아침에 결과를 받아볼 수 있게.
nodemailer smtp
EX · 03
TeamsNotifier
Microsoft Teams Incoming Webhook 으로 같은 이벤트를 방으로. 팀 단위 가시성 — 누가 무엇을 만들고 있는지 옆자리에서 안다.
teams webhook adaptive card
EX · 04
sessionSummarizer
세션 첫 prompt 가 끝나면 별도 claude 콜로 세션 제목을 자동 생성. "버튼 색 변경" / "OAuth 추가" 식으로 — 사이드바에 의미가 보이게.
title gen async
EX · 05
deployManager
메이커 본인의 배포 트리거. 코드 변경 push 시 파이프라인 → 메이커 자체가 자기를 재배포. 셀프 호스팅의 자생력.
self-deploy pipeline trigger
EX · 06
preview route
세션이 빌드한 SPA 의 dist/ 를 /preview/<project>/ 로 미리보기. referer 기반 절대경로 fallback 까지 — 라우팅이 깊어도 깨지지 않게.
preview spa fallback
OPS
반쯤은 본체, 반쯤은 운영 도구. 메이커가 살아 있게 하는 건 결국 — 알람과 리포트와 self-deploy.
15 · ADMIN CONSOLE

관리자 콘솔 — 모두 보이고, 모두 남는다

메이커는 /admin 아래 별도 SPA를 띄웁니다. 디바이스 토큰 + JWT 이중 게이트로 진입하고, 그 뒤로는 모든 prompt 가 흔적을 남깁니다.
DEVICE TOKEN · 한 번 잠그기
관리자만의 기기 화이트리스트
// 1. 최초 등록 — secret 1회 입력
POST /api/admin/activate
{ "secret": "$ADMIN_BOOTSTRAP_SECRET" }
  ↓
{ "deviceToken": "<uuid>" }   // localStorage 저장

// 2. 이후 모든 admin 요청 헤더에
GET /api/admin/online
X-Device-Token: <uuid>
Authorization: Bearer <jwt>
  ↓
device-tokens.json 매칭 실패 시 403
secret 이 새어도 등록된 기기에서만 콘솔이 열립니다. 토큰은 파일 기반 — 회수도 한 줄.
3-TIER · 권한 매트릭스
역할별로 행위를 끊어둔다
admin · 세션 전체 / prompt ✓ / users ✓ / reports ✓
user · 자기 세션 / prompt ✓ / users ✗ / reports ✗
viewer · 자기 세션 / prompt ✗ / 읽기 전용
  • · WRITE_ACTIONS 셋에 들어가면 ws 핸들러 진입부에서 자동 거부
  • · protected · navskh 계정은 삭제 불가 — 마지막 안전핀
AUDIT · 01
/api/users
사용자 관리
CRUD · 역할 변경(admin/user/viewer) · 프로젝트 접근 화이트리스트. 비밀번호 초기화·disabled 플래그 시 활성 JWT 전부 폐기해 즉시 로그아웃.
action: create | update | delete disabled flag project gate
AUDIT · 02
/api/admin/online
접속자 모니터링
activeClients 셋 그대로 노출 — 인증된 모든 WS 연결을 사용자·역할·접속 시각·현재 활성 세션 ID로 본다. 10 초마다 자동 갱신.
activeSessionId auto-refresh 10s attach / detach 동행
AUDIT · 03
/api/admin/logs
대화 기록 감사
모든 prompt/response 가 conversation_logs 에. 사용자·프로젝트·날짜로 필터 → 페이지네이션 → 행 클릭하면 프롬프트 원문·응답 요약·토큰·소요시간·툴 사용 횟수까지 펼침.
pageSize: 20 turn · tokens · duration tool count
AUDIT · 04
sessionManager
global broadcast
setGlobalBroadcast 한 줄에서 분기 — 일반 사용자는 자기 세션만, admin 에겐 모든 세션 이벤트가 흐른다. 별도 모니터링 통로가 필요 없습니다. 
if (client.userRole === 'admin'
 || client.userId === ownerUserId) {
  client.ws.send(payload);
}
SCOPE · TRUST
AI 가 자유롭게 일하는 만큼, 관찰 가능성 위에서만 그 자유가 안전합니다 — 모든 prompt 가 흔적이고, 모든 세션이 들여다보입니다.
16 · INSIGHT & OPS

분석·리포트·운영까지

본 적도 없고 보고도 안 받는 AI 작업은 그냥 비용일 뿐 — 콘솔은 그것을 읽을 수 있는 그림으로 만들어줍니다.
OPS · 01
사용자별 활동 분석
NOISE_PATTERNS 필터로 테스트·짧은 요약을 걸러내고, 의미 있는 작업만 골라 사용자 카드로. 카드마다 💡 Focus — "최근에 진짜로 뭘 만들었나" 한 줄 요약.
days: 7 / 30 / 90 / all project-grouped
OPS · 02
토큰 대시보드
Recharts 로 세 가지 시점: ① 사용자별 Bar ② 프로젝트별 Bar ③ 일자별 Line (토큰·프롬프트 dual-axis). K/M 단위 자동 포매팅으로 한눈에 비교.
group_by: user | project | date recharts
OPS · 03
일일 리포트
scheduler 가 매일 자동 — 카드에 📄 작업요약 · 💾 커밋 수 · 🚀 배포 대기 · 🗄️ DB 마이그레이션 표시. 마이그레이션은 주황 강조 (사람 손이 필요한 항목).
daily / manual conversationStats · gitActivity pendingMigrations
OPS · 04
배포 상태 SSE
/api/deploy/status/stream 으로 실시간 스트리밍. 첫 응답에 캐시된 상태 + 백그라운드로 fresh 조회가 들어와 누적 push. 사용자는 polling 없이 라이브로 본다.
EventSource cached → fresh per project
OPS · 05
promote / sync
관리자 전용 두 버튼. promote · 알파에서 안정화된 코드를 main 으로 승격. sync · main 핫픽스를 다시 알파로 역동기화. 메이커가 직접 git workflow 를 자동화.
alpha → main main → alpha force option
OPS · 06
서비스 재시작
"재시작" → "확인" 의 두 번 클릭 안전장치. SIGTERM 으로 graceful 종료, 진행 중 세션 정리, PM2 가 다시 띄움. UI 는 5 초 후 자동 새로고침.
POST /api/system action: restart PM2 respawn
OBSERVABILITY
쓴 만큼 보이고, 본 만큼 조정할 수 있게. 콘솔이 곧 메이커의 운전석.
